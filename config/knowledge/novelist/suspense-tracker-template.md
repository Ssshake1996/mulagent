# 悬念追踪器模板

用于跟踪小说中所有层级的悬念/伏笔/钩子。

---

## 悬念生命周期

每个悬念经历四个阶段：

```
planted（埋设）→ fermenting（发酵）→ revealed（揭示）→ closed（闭合）
```

| 阶段 | 含义 | 操作 |
|------|------|------|
| **planted** | 首次引入线索或暗示，读者可能注意到也可能没注意到 | 记录埋设章节 |
| **fermenting** | 后续章节中多次强化、暗示、加深读者好奇心 | 记录每次推进的章节 |
| **revealed** | 真相揭露，产生"原来如此"的冲击 | 记录揭示章节 |
| **closed** | 揭示后的影响消化完毕，该悬念线彻底结束 | 记录闭合章节，归档 |

---

## 四级悬念层次

### 第一级：全书悬念（Book-Level）

跨越整本书的核心谜题。通常只有1-3条。整本书结束时才完全揭示。

| ID | 悬念描述 | 状态 | 埋设章节 | 发酵章节（关键节点） | 揭示章节 | 闭合章节 |
|----|---------|------|---------|-------------------|---------|---------|
| B-01 | | planted | 第X章 | | | |

**示例：**
- "主角的父亲为何消失" — 贯穿全书
- "世界的真正运行法则是什么" — 最终章揭示
- "反派与主角的真实关系" — 大结局反转

### 第二级：卷级悬念（Volume-Level）

跨越一卷的悬念。每卷2-5条。卷末应当揭示或闭合。

| ID | 所属卷 | 悬念描述 | 状态 | 埋设 | 发酵 | 揭示 | 闭合 |
|----|-------|---------|------|------|------|------|------|
| V1-01 | 卷一 | | planted | | | | |

**示例：**
- "谁是幕后操控者" — 本卷大BOSS揭露
- "主角的新能力有什么副作用" — 卷末爆发

### 第三级：篇章悬念（Arc-Level）

跨越一个篇章（10-30章）的悬念。每篇章1-3条。

| ID | 所属篇章 | 悬念描述 | 状态 | 埋设 | 发酵 | 揭示 | 闭合 |
|----|---------|---------|------|------|------|------|------|
| A1-01 | 卷一/篇章一 | | planted | | | | |

**示例：**
- "密室里的尸体是谁" — 篇章末揭晓
- "盟友的真实目的" — 3-5章内揭示

### 第四级：章节钩子（Chapter-Level）

单章结尾的即时悬念。每章1-2条。通常在下一章开头回应。

| ID | 所属章节 | 钩子描述 | 已回应 | 回应章节 |
|----|---------|---------|--------|---------|
| C001-01 | 第001章 | | 否 | |

---

## 悬念健康度规则

在任意时间点，小说应当满足：

### 活跃悬念数量
- 至少1条全书悬念处于 **fermenting**
- 至少1-2条卷级悬念处于 **fermenting**
- 当前篇章至少1条篇章悬念处于 **planted** 或 **fermenting**
- 每章结尾至少1条章节钩子

### 新旧平衡
- 每章至少回应或推进1条旧悬念
- 每章至少埋设或强化1条悬念
- 避免同时揭示3条以上悬念（信息过载）

### 时效检查
- planted 超过 **20章** 未进入 fermenting → 警告：读者可能已遗忘
- fermenting 超过 **50章** 未 revealed → 警告：拖延过久
- 章节钩子超过 **3章** 未回应 → 错误：必须处理

### 级联规则
- 高级悬念揭示时，应触发或解决多条低级悬念
- 全书悬念的每次"发酵"应伴随至少一条卷级/篇章级悬念的推进
- 卷级悬念闭合时，检查是否有相关篇章悬念遗漏未闭合

---

## 悬念审计检查清单

每10章或篇章结束时执行：

- [ ] 是否有 planted 超过20章未发酵的悬念？
- [ ] 是否有 fermenting 超过50章未揭示的悬念？
- [ ] 是否有章节钩子超过3章未回应？
- [ ] 当前活跃的 fermenting 悬念是否足够多（至少3条）？
- [ ] 最近10章内是否有悬念揭示产生了"原来如此"效果？
- [ ] 各级悬念的揭示节奏是否递进（越到后期揭示越重磅）？
- [ ] 卷末是否所有卷级悬念都已 revealed 或 closed？
- [ ] 是否有任何悬念在揭示前缺少足够的发酵铺垫（至少2次发酵）？

---

## 悬念归档

状态变为 **closed** 的悬念，移动到文件末尾的"已闭合悬念归档"区域，保持活跃区域精简。

```markdown
---

## 已闭合悬念归档

### [ID] [悬念描述]
- 埋设：第X章
- 发酵：第X、X、X章
- 揭示：第X章
- 闭合：第X章
- 回顾：[一句话总结这条悬念线的完整弧光]
```

---

## 标准模式（≤50章）简化用法

50章以下的小说不需要四级结构。简化为两级：

### 主线悬念
| ID | 悬念描述 | 状态 | 埋设 | 发酵 | 揭示 |
|----|---------|------|------|------|------|

### 章节钩子
| 章节 | 钩子描述 | 已回应 |
|------|---------|--------|

仍然遵循健康度规则中的新旧平衡和时效检查。
