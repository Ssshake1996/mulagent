# 多Agent协作框架 - 架构设计

## 技术选型

| 层级 | 组件 | 技术选型 | 选型理由 |
|------|------|----------|----------|
| **用户接口** | API网关 | FastAPI | 异步高性能，原生WebSocket支持 |
| | Web UI | Streamlit / Gradio | 快速原型，后期可换React |
| | 消息平台 | OpenClaw Gateway | 原生支持15+平台适配 |
| **调度层** | 意图识别 | OpenClaw Agent | 利用其ReAct推理循环 |
| | 模板匹配 | Qdrant语义检索 | 向量相似度匹配历史案例 |
| **编排层** | 任务DAG | LangGraph | 专为LLM工作流设计的DAG引擎，支持条件分支、循环、人工干预节点 |
| | 执行控制 | Celery + Redis | 成熟的分布式任务队列，支持重试、超时、优先级 |
| | Agent间通信 | Redis Streams | 低延迟，支持消费者组，天然支持事件溯源 |
| **Agent池** | Agent运行时 | OpenClaw | 本地优先，Markdown记忆，技能插件机制 |
| | Agent注册 | 自定义Registry | 基于OpenClaw的多Agent路由扩展 |
| **记忆系统** | 会话记忆 | Redis | 快速读写，TTL自动过期 |
| | 长期记忆 | OpenClaw Markdown + Qdrant | OpenClaw原生文件记忆 + 向量检索 |
| | 结构化存储 | PostgreSQL | 案例库、指标、模板等结构化数据 |
| **工具层** | 代码沙箱 | Docker | 安全隔离执行环境 |
| | 浏览器 | Playwright | 现代浏览器自动化 |
| | 搜索 | Tavily / SerpAPI | 专为AI优化的搜索API |
| **能力获取** | 历史经验匹配 | Qdrant语义检索 | 从案例库中匹配已验证的方案 |
| | Web知识搜索 | Tavily / SerpAPI | Agent基于搜索知识自主推理 |
| | 外部Skill引入 | OpenClaw Skills格式 | 可执行能力，需安全审查 |
| **安全审查** | 静态分析 | semgrep + bandit | Python安全扫描行业标准工具 |
| | 权限评估 | 自定义规则引擎 | 最小权限校验 + 同类基线对比 |
| | 沙箱试运行 | Docker SDK | 隔离执行 + 行为监控 |
| | 运行时监控 | 自定义 + Prometheus | 异常行为检测 + 信任度动态调整 |
| **进化层** | 经验提取 | OpenClaw Agent | 专门的"元Agent"负责模式抽象 |
| | 案例库 | Qdrant + PostgreSQL | 语义检索 + 结构化查询 |
| | 策略优化 | 自定义（基于统计） | 初期基于规则和统计，后期可引入强化学习 |
| **基础设施** | 监控 | Prometheus + Grafana | 行业标准，丰富的生态 |
| | 日志 | structlog | Python结构化日志，JSON输出 |
| | 配置 | YAML + openclaw.json | OpenClaw原生配置 + 自定义扩展 |

## Agent能力获取与Skill安全机制

### 核心原则：三级学习路径，风险递增

当Agent执行任务时发现能力不足，按以下优先级逐级尝试：

```
优先级1：历史经验复用（零风险）
    │ 查案例库，有匹配直接复用验证过的方案
    │ 无匹配 ↓
    ▼
优先级2：Web搜索获取知识（低风险）
    │ 搜索获取的是信息/知识，Agent基于知识自主推理
    │ 信息不足以解决，需要可执行能力 ↓
    ▼
优先级3：外部Skill引入（高风险，需安全审查）
    │ 进入Skill安全审查管线
    ▼
```

### 外部Skill的威胁模型

| 威胁类型 | 描述 | 危害等级 |
|----------|------|---------|
| 数据泄露 | Skill中嵌入HTTP请求，静默上传用户上下文/文件/密钥 | 严重 |
| Prompt注入 | SKILL.md描述文件包含诱导指令，劫持Agent行为 | 严重 |
| 权限滥用 | 声称功能A，实际请求远超必要的权限 | 高 |
| 供应链污染 | 热门Skill被作者在更新中注入恶意代码 | 高 |
| 质量陷阱 | Skill无恶意代码，但方法论/逻辑错误，产出有害结果 | 中（更隐蔽） |

### Skill安全审查管线（4阶段）

```
外部Skill候选
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│  Stage 1: 静态分析                                       │
│  ├─ AST扫描：网络请求、文件操作、系统调用、eval/exec      │
│  ├─ 权限声明 vs 实际代码行为一致性校验                     │
│  ├─ SKILL.md prompt注入检测（已知注入模式匹配）            │
│  └─ 已知恶意签名库匹配                                    │
│  工具：semgrep / bandit（Python安全扫描）                  │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│  Stage 2: 权限评估                                       │
│  ├─ 最小权限原则：Skill真的需要这些权限吗？               │
│  ├─ 同类Skill典型权限基线对比                             │
│  └─ 超出必要权限 → 标记可疑，降低信任评级                  │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│  Stage 3: 沙箱试运行                                     │
│  ├─ 在隔离Docker容器中执行（无外部网络/只读FS）           │
│  ├─ 监控：网络流量、文件访问、资源消耗                     │
│  ├─ 用已知答案的测试输入验证输出正确性                     │
│  └─ 记录完整行为日志供审计                                │
│  工具：Docker SDK + 自定义监控探针                        │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│  Stage 4: 信任评级                                       │
│  ├─ ✅ trusted    — 全部通过，可在Agent进程内直接使用      │
│  ├─ ⚠️  sandboxed — 部分可疑，仅允许在沙箱中运行          │
│  ├─ 🔍 human_review — 高风险项，暂存等待人工审批          │
│  └─ ❌ blocked    — 明确恶意，拒绝并上报                   │
```

### 运行时隔离策略

| 信任等级 | 运行环境 | 网络访问 | 文件系统 | 资源限制 |
|---------|---------|---------|---------|---------|
| trusted | Agent进程内 | 正常 | 正常 | 无 |
| sandboxed | 独立Docker容器 | 禁止或白名单域名 | 只读 + 临时写入区 | CPU/内存/时间配额 |
| human_review | 不运行 | - | - | - |
| blocked | 不运行 | - | - | - |

sandboxed Skill的所有输出经过内容清洗后再传回Agent，防止输出中携带注入指令。

### Skill质量验证（安全之外的第二道关）

安全审查保证"不作恶"，质量验证保证"做得好"：

- **效果验证** — 用已知答案的benchmark测试用例实际运行，评估输出质量
- **对比基线** — Skill辅助的结果 vs Agent纯推理的结果，无显著提升则不引入（纯增加攻击面）
- **持续监控** — 每次执行记录行为日志 + 异常检测（如突发大量网络请求）
- **信任度动态调整** — 表现好 → 提升信任等级；异常/差评 → 降级直至移除
- **更新强制重审** — Skill版本更新时，必须重新走完整审查流程

### Skill来源分级

| 来源 | 审查力度 | 说明 |
|------|---------|------|
| 官方内置 | 免审查 | 系统预置的Skill，完全可信 |
| 认证社区源 | 轻量审查（Stage 1+2） | 经过社区审核的Skill仓库 |
| 野生源 | 完整审查（Stage 1-4） | 任意来源，必须走完全部流程 |

### Phase策略

- **Phase 1**：保守方案 — 所有外部Skill必须人工审批，系统只做预审和推荐
- **Phase 2**：半自动 — 认证源Skill审查通过后自动启用，野生源仍需人工
- **Phase 3**：全自动 — 基于积累的审查案例训练自动化审查模型，仅高风险上报人工

## 开发路线

### Phase 1：跑通端到端流程

1. **基础设施搭建** — docker-compose (Redis + PostgreSQL + Qdrant) + 配置加载
2. **Agent池** — base_agent封装OpenClaw + 注册中心 + 1个预置Agent（代码Agent）
3. **编排层核心** — DAG引擎（先支持串行流水线）+ 简单选派（技能匹配）+ 基础执行控制
4. **调度层** — 意图识别Agent + 简单分类（直接路由，暂不用模板匹配）
5. **Gateway** — FastAPI + 基础WebSocket
6. **记忆** — 会话记忆(Redis) + OpenClaw原生长期记忆
7. **能力获取（最简版）** — 历史经验复用 + Web搜索知识（暂不引入外部Skill）
8. **进化层（最简版）** — 反馈收集 + 轨迹记录（先只存，不做经验提取）

### Phase 2：安全机制 + 进化闭环

1. **Skill安全审查管线** — 静态分析(semgrep) + 权限评估 + 沙箱试运行 + 信任评级
2. **外部Skill引入** — 先开放认证社区源（轻量审查），所有Skill需人工审批
3. **质量验证** — 效果测试 + 基线对比 + 运行时监控
4. **进化层完整版** — 经验提取 + 案例库 + 模板自动生成 + 技能缺口分析

### Phase 3：自动化进化 + 规模扩展

1. **审查自动化** — 基于积累案例训练自动审查模型，仅高风险上报人工
2. **开放野生Skill源** — 完整4阶段审查流程
3. **编排策略自优化** — 选派权重自动调整 + 分解策略偏好学习
4. **Agent自生成** — 元Agent根据技能缺口自动创建新Agent配置
